
# Natick CNY 2026 - Backend Architecture & Schema

**Status:** Active / Implemented
**Backend:** Google Firebase (Firestore)

This document outlines the current data structure and security model used in the production system. The system has been migrated from `localStorage` to **Firestore**.

---

## 1. Architecture Overview

The application uses a serverless architecture relying on Firebase features:
*   **Firestore**: Primary database for reservations and system config.
*   **Authentication**: Used for Staff/Admin access control (via Access Codes mapped to Roles).
*   **Analytics**: Event tracking (GA4).

---

## 2. Firestore Schema

### Collection: `reservations`
Stores all attendee data.
**Document ID:** Auto-generated by Firestore (e.g., `7f8a9s8d7f`).

| Field | Type | Description |
| :--- | :--- | :--- |
| `id` | String | Public ID (e.g., "CNY26-1234"). Indexed for search. |
| `createdTime` | Timestamp | Creation date. |
| `contactName` | String | Full name. Indexed for search. |
| `phoneNumber` | String | Digits only. Indexed for search. |
| `email` | String | Contact email. |
| `ticketType` | String | `EarlyBird` \| `Regular` \| `WalkIn` |
| `adultsCount` | Number | Number of adults. |
| `childrenCount` | Number | Number of children (free). |
| `totalPeople` | Number | `adultsCount` + `childrenCount`. |
| `totalAmount` | Number | Expected revenue. |
| `paidAmount` | Number | Collected revenue. |
| `paymentStatus` | String | `Unpaid` \| `Paid` |
| `checkInStatus` | String | `NotArrived` \| `Arrived` \| `Cancelled` |
| `isPerformer` | Boolean | If the registrant is a performer. |
| `performanceUnit` | String | Name of the performance group. |
| `lotteryNumbers` | Array<String> | Assigned at check-in (e.g., `["101", "102"]`). |
| `operatorId` | String | Who created/modified the record (e.g., 'PUBLIC' or Staff Code). |

### Collection: `system`
Singleton configuration documents.

**Document: `ticketConfig`**
Manages inventory caps.
```json
{
  "totalCapacity": 400,
  "earlyBirdCap": 300,
  "regularCap": 50,
  "walkInCap": 50
}
```

### Collection: `mail` (Trigger Extension)
Used to trigger email sending via Firebase Extensions (e.g., "Trigger Email from Firestore").
*   **to**: `[email]`
*   **message**: `{ subject: "...", html: "..." }`

### Collection: `access_keys`
Maps simple access codes to roles for the login system.
**Document ID:** The 6-character code (e.g., `A9X2M1`).

```json
{
  "role": "admin" // or "staff"
}
```

## 3. Security Rules (Reference)

Recommended Firestore Security Rules:

```javascript
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {
    
    // Helper function
    function isStaff() {
      // In a real auth scenario with Custom Claims
      // return request.auth.token.role == 'staff' || request.auth.token.role == 'admin';
      return request.auth != null; 
    }

    // Public Registration
    match /reservations/{docId} {
      // Allow public to create
      allow create: if true;
      
      // Allow public read only for their own lookup (enforced by query constraints in app)
      // Ideally, secure this further in production
      allow read: if true; 
      
      // Only staff can update/delete
      allow update, delete: if isStaff();
    }
    
    // System Config
    match /system/{docId} {
      allow read: if true;
      allow write: if isStaff();
    }
    
    // Mail Collection
    match /mail/{mailId} {
      allow create: if true; // Allow system to send emails on public registration
    }
    
    // Access Keys
    match /access_keys/{code} {
      // Used by login service to validate code
      allow read: if true;
      allow write: if false; 
    }
  }
}
```

## 4. Key Workflows

1.  **Registration**: Client creates doc in `reservations` -> Cloud Function/Extension triggers Email.
2.  **Check-in**: Staff searches by `phoneNumber` -> Updates `checkInStatus` to `Arrived` -> Generates `lotteryNumbers`.
3.  **Stats**: Admin Dashboard queries full collection (client-side aggregation) to calculate totals.
